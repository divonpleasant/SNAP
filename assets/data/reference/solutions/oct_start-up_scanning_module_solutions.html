These errors occur when any of the start-up diagnostic tests fail as the software launches. The customer-presented error is generic and could have many underlying root causes. Regardless of root cause, the instrument cannot Acquire scans until or unless this error is resolved.

A hard power cycle should always be attempted first before any deeper investigation. Often the diagnostic failure will be transient and a reboot will allow the device to start up without any further errors. If the issue is intermittent but persisent, it may require an FSE onsite dispatch to confirm the hardware components are not compromised.

Details

To determine the root cause, click on the Details button to the left of the Continue button, then scroll to the bottom to find the message marked with "FAIL," and cross reference with the common messages table, below:

Error Message Recommended Action
  • Object reference not set to an instance of an object
  • FW initialization failed, most likely controller is disconnected
  • Confirm Windows login account has local administrator permissions
  • Confirm LAN 2 (Internal NIC) is enabled and properly configured
  • Check available disk space on C: drive
  • Check Framegrabber driver: click Help > About and note the Framegrabber driver version; if it shows a question mark ("?"), reinstall the Coreco driver
  • Verify there are not two instances of the software running. You may not see the error dialog which says, "Another instance of this program is running. Please close before continuing" unless you log off the primary screen instance
  • Galvo test failed (for X, Y, and/or L galvo)
  • If the standard power cycle does not work, occasionally running a manual diagnostic test can exercise the galvos sufficiently to allow the start up tests to pass
  • If restart and manual diagnostics do not resolve the issue, root cause could be the galvo(s), power conversion board, framegrabber, or the micro-controller. Dispatch an FSE for onsite evaluation
  • Power supply out of range (for one or more voltage power supplies)
  • Check software version. If version is 7.0, this is an intermittently occurring known issue. Upgrade the software to 7.5+
  • If software is 7.5+ already and the problem persists, it may be a failing power supply or power conversion board. Dispatch an FSE for onsite evaluation
  • OCT camera initialization failure. Camera pixel clock is not detected
  • Check the software version and Microsoft .NET Framework. Older versions of the OCT software (6.5   7.0 with .NET 4.5.1+, fixed in 7.5) had a known issue requiring a .NET downgrade
  • If upgrading does not resolve the issue, the root cause may be due to a loose power connector to the OCT camera or a defective spectrometer. Dispatch an FSE for onsite evaluation
  • LSLO camera initialization failure. Camera pixel clock is not detected
  • If software version is 9.0, especially if the issue is intermittent, determine whether the device is being rebooted frequently or if the error occurs mostly when the device is rebooted warm. This could be due to a known issue. The workaround requires a cold boot (shut down the instrument and allow the system to rest for 1-5 minutes before restarting)
  • If the software version is 9.5+ and a cold boot does not resolve the issue, it could be a loose power connector to the LSLO or a defective LSLO camera. Dispatch an FSE for onsite evaluation
  • Stored calibration values are not the same as specified
  • Download the calibration file from the microcontroller to the hard drive (see the Service Manual, Section 4.2.17.2: Sending Calibration Data to the Hard Disk)

If the root cause is not listed above, or if the problem persists after checking connections and power, a Field Service Engineer onsite dispatch is required to resolve the issue.
